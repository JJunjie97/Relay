# USEEngine 波形输出控制方式 (V3 架构)

本文档阐述 `USEEngine` (V3) 底层执行机构的实现机理。在 V3 全新架构下，底层引擎彻底抛弃了 `Start`/`Stop` 生命周期与运行时编译，化身为一个**零编译、完全依靠节点跳转驱动的永久状态机 (FSM)**。

---

## 1. 核心架构：死循环与控制权接管

### 1.1 零生命周期模型
引擎在启动后即进入无限死循环 (`coreLoop`)。它不再有“停止状态”，所谓的待机或停止，仅仅是停留在特定的**内置功能节点**上：
*   **`Node 0x0000` (待机节点)**：模式 1 (Static)，下发硬件防抖配置与 `START` 启动波形发生器。引擎在此节点由于默认 1000 秒的超时而处于长时循环等待（每 1000 秒刷新一次）。
*   **`Node 0xFFFF` (终止节点)**：模式 1 (Static)，下发硬件 `RESET` 重置所有缓冲区并切断输出。
*   **生命周期转移**：外部系统 (如 `TestCtrl`) 仅需调用 `engine.manualTrig(firstNodeId)` 即可从待机跳入测试节点启动波形；调用 `engine.manualTrig(0xFFFF)` 即可触发急停进入终止节点。

### 1.2 零底层编译的 `USENode`
所有的波形数据均被前置到了 API 模块层。底层直接接收预先编译好的二进制帧 (`bytes`) 数组：
*   `baseFrame`: `WR_SHADOW` 帧数组（基准帧）
*   `stepFrames`: `STEP_SHADOW` 帧的二维矩阵（非均匀步进）
*   `gateFrames`: `PHASE_GATE` 帧矩阵（门控阻塞指令）
*   `resetFrame`: `WR_STAGE` 帧数组（复归态隔离帧）

底层除了直接调用 `flush()` 向串口推流，**不包含任何解析计算逻辑**。

---

## 2. 四大执行模式 (Execution Modes)

这四种模式对应着继电保护测试仪极其复杂的波形变化需求，并且完全利用了底层的基准 (Base)、影子 (Shadow) 缓冲倒换。

### 2.1 Mode 1: Static (静态稳态)
用于输出一段完全不变化的恒定波形（如：故障前的准备态）。
*   **核心动作**：发送 `baseFrame` (将参数写至 Base+Shadow) -> 发送 `SYS_UPDATE` (推入 Active 层物理生效)。
*   **延时驱动**：挂起等待直到 `timeout` 到期或被外部触发打断。

### 2.2 Mode 2: Sweep (非均匀步进)
用于输出随时间阶梯变化的波形（如：频率滑差、均匀增量）。在 V3 中，由于 `stepFrames` 为二维矩阵，它可以支持每个 Tick 不同的非线性步进量。
*   **帧机制**：首拍对齐发出 `baseFrame` 并生效。随后进入循环，每一个 Tick 开始时将对应的 `stepFrames[N]` 推入 FPGA (这只会进入 Shadow 缓冲挂起)，然后 `sleep` 直到 Tick 时间点到达，再发出 `SYS_UPDATE` 令其瞬间同时生效。

### 2.3 Mode 3: Reset (复归态/断开状态交替)
用于模拟如多次重合闸、瞬时故障等“故障-恢复-故障”交替出现的波形。每个正常的故障 Tick 被拆分为两个阶段：输出期和复归期。
*   **隔离策略 (极其核心)**：
    由于复归期的波形（如电压恢复额定，电流归零）可能与发生故障期的基准波形完全不同，引擎采用了 `SYS_WR_STAGE` (仅写影子层) 的硬核隔离手段。
    `baseFrame` 将故障基准写入 `Base`。
    `resetFrame` 将隔离期的恢复值以 `WR_STAGE` 写入 `Shadow`。
    这样一来，当我们需要在恢复期和故障期来回切时，`Base` 层一直留着纯粹的故障基底值不被污染，从而保证后续加上 `stepFrames` 进行故障递增时，计算锚点永远是对的。
*   **急停响应与假死防护**：复归等待期极长（可能有几十秒），因此底层在等待时使用了特制的 `sleepWait` 方法，屏蔽所有的普通 DI 触发动作，只为 `manualTrig` 留下“逃生通道”，保障紧急切断绝对不卡顿。

### 2.4 Mode 4: DcComp (直流补偿/指定相位换流)
这是整个继保系统要求最高的模式。需要等波形运行到指定相位（例如某相电压过零点）时，瞬间改变直流分量进行衰减，实现极高精度的变压器合闸涌流模拟。
*   **FIFO 硬件排队与阻塞效应**：
    发送出去的 `gateFrames[N]` 是 `SYS_PHASE_GATE` 门控命令。如果当前的相位没有达到预设值，这个命令会**卡死**在 FPGA 内部的执行 FIFO 队列头上。
    此时，引擎可以肆无忌惮地把后续要执行的 `SYS_SYNC`（强制覆盖刷新）和下一拍的步进帧紧紧跟在门控指令后面一股脑 `flushNoAck` 出去。
    因为被前头的门控指令堵着，它们绝对不会提前执行；而一旦相位吻合门控放行，积累在 FIFO 里的所有更新命令会在纳秒级别立刻灌入 Active 层，做到纳秒级零等待倒换！

## 3. CoreLoop 统一清理与垃圾回收 (GC)

### 3.1 统一 `Teardown` 清洗
为了解决在复杂步进或复归中断时留下脏数据的问题，V3 引擎将清理逻辑剥离出了各个运行模式，集中于 `coreLoop` 中：
```python
await self.flush([HWCodec.FRAME_SYS_SYNC])
```
无论上次跳转是在什么节点、什么状态下发生的，**只要在进入新节点的第一步，先无脑发送一条 `SYS_SYNC` 指令**。这会将上一次执行可能残留在 Active 层的值反向覆写回 Base 和 Shadow 层，彻底洗净过去，让新节点的计算有一个纯净同步的起点。

### 3.2 节点垃圾回收
为防止内存泄漏，每当 `nextId` 跳转至 `0xFFFF` 时（即测试完整结束并触发重置时），`coreLoop` 会自动切断只保留系统核心节点 (0x0000 和 0xFFFF)，清空在此之前内存中驻留的所有动态波形对象。这也要求了前端或中转层 (TestCtrl) 必须在下一次 `Start` 前显式进行 `upsertNodes` 操作。
