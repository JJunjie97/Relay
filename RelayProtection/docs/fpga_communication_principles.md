# FPGA 通讯原理与底层波形发生机制说明

本文档基于 `documentation-and-adrs` 技能规范编写，旨在作为系统级架构说明（Architecture Decision Records）的一部分。文档详细阐述了软硬件之间的指令封装结构、所有的系统/参数指令表、DDS 波形生成机制，并**重点强调了在开发中极易引发致命错误的数据类型边界及参数解析陷阱**。

---

## 1. 核心重点警示：极易踩坑的参数解析逻辑

> [!CAUTION]
> **红线警告：不同层级 (Index) 的参数意义完全不同！**

在向 FPGA 写入波形参数时，负载带有两个 32位数据：`p1` 和 `p2`。但它们的物理意义和位宽处理方式**完全取决于你正在写入哪个层级 (Index)**：

*   **常规交流层 (Index `0x00` ~ `0x3F`)**：
    *   **p1 = 幅度 (Amplitude)** 
    *   **p2 = 相位 (Phase)** 
*   **直流层 (Index `0xFF`)**：
    *   **p1 = 幅度 (Amplitude)** 
    *   **p2 = 频率 (Frequency)** *(注意：这里不再是相位！)*

> [!WARNING]
> **数据截断与精度陷阱 (Precision Truncation)**
> 
> *   **p1 (幅度)**：上位机（Python）传入的是 32 位无符号整数 (`U32`)，但协议层**只会取其高 16 位**发给 FPGA。FPGA 收到后，会直接将其作为 **`I16` (16位有符号整数)** 处理！这就是为什么在代码里电压和电流的最大极限值通常不能越过 32767，否则就会变成负数。
> *   **p2 (频率/相位)**：上位机传入完整的 `U32`。如果当前是直流层（此时 p2 是频率），FPGA 会取满 **32位作为 `U32`** 处理；但如果当前是交流层（此时 p2 是相位），FPGA 会**仅截取高 16 位作为 `U16`** 进行环形处理（满 `0xFFFF` 绕回）。

## 2. 三级缓冲区工作机制 (Three-Tier Buffer System)

为了保证波形输出的平滑切换以及在更新多个通道/谐波参数时不产生信号毛刺，FPGA 内部采用了 **Base（基准）** -> **Shadow（影子）** -> **Active（活动输出）** 的三层流水线缓冲架构。

*   **Base 层（基准缓冲）**：底层参数的“记忆库”。**任何写入操作都会首先覆盖 Base 中的相关通道与层级的值**。所有的 Step（步进/加法）操作，都是读取 Base 中的历史值作为被加数。
*   **Shadow 层（影子缓冲）**：预加载缓冲。用于挂起即将生效的参数集合，等待同步更新信号（Update/Ping-Pong）到来后一次性“倾倒”。
*   **Active 层（活动缓冲）**：这是**直接决定 DAC 实时输出波形**的物理寄存器。只能通过指令驱动将前面的数据覆盖进来，不能直接算术干预。

---

## 3. 完整指令集解析 (Command Instruction Set)

硬件支持两种类型的通信帧：**SYS（系统控制帧，以 `0x5A` 打头）** 和 **PARAM（参数写入帧，以 `0xA5` 打头）**。

### 3.1 核心通信假定 (Critical Communication Assumption)

> [!IMPORTANT]
> **任何下发命令均会回复 ACK**
> FPGA 对上位机发送的每一条物理帧（无论是 SYS 还是 PARAM 帧），在成功接收并执行完毕后，都会向串口回复一个包含 8 字节的 ACK 帧（带当前硬件时间戳）。
> `USEEngine` 正是依据 `sentCount` 和 `ackCounter` 的绝对一致性来确认所有命令在 FIFO 队列中的完成进度的。

### 3.2 系统指令 (SYS Commands)
主要用于控制发生器的状态机、滤波器配置和触发状态转移。

| 指令代码 (Hex) | 指令名 | 描述说明 |
| :--- | :--- | :--- |
| `0x00` | **SYS_START** | 启动波形发生器输出。 |
| `0x01` | **SYS_STOP** | 停止发生器输出，输出置零。 |
| `0x04` | **SYS_RESET** | **全局硬件硬复位**，清空所有缓冲区并复位所有参数。 |
| `0x05` | **SYS_UPDATE** | **触发 Ping-Pong 倒换**：DDS 读取参数的缓冲区调换。将 Shadow 缓冲推送到 Active 层，实现多通道波形的同步更新。 |
| `0x20` | **SYS_SET_DBNC** | 设置数字输入 (DI) 的防抖滤波时间。参数 `p_U8` 表示滤波时长 (每单位 = 0.32768ms @ 50MHz)。 |
| `0x21` | **SYS_SET_DO** | 直接写入数字输出 (DO) 状态掩码，控制物理继电器或光耦。 |

### 3.2 DDS 写入指令 (Write Commands)
这些指令用于**覆盖**现有的波形参数（幅度、相位、频率）。

| 指令代码 (Hex) | 指令名 | 描述说明 |
| :--- | :--- | :--- |
| `0x10` | **DDS_WR_SHADOW** | **影子写入**：写参数到 `Base` + `Shadow`。波形维持原样，需配合后续的 `SYS_UPDATE` 指令才能生效。 |
| `0x11` | **DDS_WR_STAGE** | **暂存写入**：写参数到 `Shadow`（`Base` 不变）。用于需要保留 Base 基准值的场景。 |
| `0x12` | **DDS_WR_LOAD** | 写参数到 `Base` + `Shadow`，随后**立刻自动触发**一次 Ping-Pong 倒换更新。 |
| `0x13` | **DDS_WR_PUSH** | 写参数到 `Shadow`（`Base` 不变），随后立即触发 Ping-Pong 倒换更新。 |

### 3.3 DDS 步进指令 (Step Commands)
这些指令不直接覆盖参数，而是将传输的参数**作为增量**，加到 `Base` 层的现有值上。

| 指令代码 (Hex) | 指令名 | 描述说明 |
| :--- | :--- | :--- |
| `0x14` | **DDS_STEP_SHADOW** | **影子步进**：`Base` = `Base` + `Step`，并将结果写入 `Shadow`（挂起等待生效）。 |
| `0x15` | **DDS_STEP_STAGE** | **暂存步进**：`Shadow` = `Shadow` + `Step`（`Base` 不变）。用于需要保留 Base 基准值的步进场景。 |
| `0x16` | **DDS_STEP_LOAD** | 执行步进计算 `Base` = `Base` + `Step`，结果写入 `Shadow` 后，自动触发 Ping-Pong 更新。 |
| `0x17` | **DDS_STEP_PUSH** | 执行步进计算 `Shadow` = `Shadow` + `Step`（`Base` 不变），自动触发 Ping-Pong 更新。 |

### 3.4 进阶控制指令

| 指令代码 (Hex) | 指令名 | 描述说明 |
| :--- | :--- | :--- |
| `0x1F` | **DDS_PHASE_GATE** | **相位门控翻转**：延迟 Update 信号的执行，直到指定的波形通道达到目标相位角度时，自动将 Shadow 切换入 Active。用于实现极高精度的过零点无缝换流等场景。 |

> [!CAUTION]
> **DDS_PHASE_GATE (0x1F) 传参特例陷阱**
> 
> 在此命令下，参数结构与其他指令**完全不同**，绝对不能将它视为普通的 Mask 操作！
> 
> 1. **`p2` 参数**：取其**高 16 位**作为更新的“参考目标相位”。
> 2. **`chMask` 参数**：在这里它**绝不是**通道掩码，而是**底层计算循环的匹配索引**！
>    *   **底层机制**：FPGA 内部运行着一个 1024 步（单次）的计算大循环（16 个通道 × 64 次谐波 = 1024）。其中 `0~63` 步是在计算 `CH0`，`64~127` 步是在计算 `CH1`，以此类推。
>    *   **比对逻辑**：你传给 `chMask` 字段的值，会被拿去和这个 `0~1023` 的内部计数器进行严格相等的比较。因此，你实际传入的必须是 `物理通道号 * 64`（即逻辑左移 6 位）。
>    *   **触发条件**：当内部计算循环刚好跑到代表该通道基频的计数值（即 `ch*64`），**并且**此刻通道内部的实时相位等于 `p2` 的高 16 位传入的目标相位时，FPGA 才会自发执行两个缓冲区的调换（Update）。
> 
> **FIFO 阻塞效应与极其精妙的同步编排**：
> 因为 `PhaseGate` 指令是下发给底层波形发生器的，如果目标相位尚未到达，**该指令会一直卡在 FPGA 的内部执行队列 (FIFO) 首部不返回 ACK**！
> 利用这一硬件特性，上位机可以将后续的 `SYS_SYNC` 甚至下一拍的波形帧紧紧跟在 `PhaseGate` 帧后面全部一次性发出，且完全不担心提前生效。因为前面的 `PhaseGate` 没触发、没 ACK，后面的 `SYNC` 就绝对没有机会执行。这是一种“让硬件替软件排队对齐”的高阶玩法。
> 
> **代码防呆建议**：强烈建议直接调用 `FPGACodec.BuildPhaseGateFrame(physical_ch, phase_u32)` 专用接口。在业务层（如 USEEngine）中，通道和相位应当以最高精度（U32）计算完毕，调用此接口时只需传入已映射好的物理通道号和全精度 32 位相位，底层将自动组装 `ch * 64` 以阻断掩码误传风险，同时 FPGA 端会自动截取相位的有效高 16 位进行比对。

## 4. 步进模式下的饱和与溢出灾难 (Step Overflow Disasters)

由于 Step 模式底层是由加法逻辑器驱动的，不同的数据类型在 FPGA 内部有着截然不同的越界处理策略：

1.  **频率 (U32 + I32)**：**饱和加法**，到达上限后维持最大值，不会绕回。
2.  **相位 (U16 + I16)**：**绕回加法**，到达 `0xFFFF` 后加一会变成 `0x0000`。
3.  **DC 直流幅度 (I16 + I16)**：**饱和加法**，到达极限值即停止增加。
4.  **交流层级幅度 (U16 + I16)**：**最危险的区域**！
    *   在底层参与 DAC 波形乘法时，这个幅度会被转化为 `I16`。
    *   **但在 Base 缓冲区中，它是作为无符号数 `U16` 存放和相加的**。
    *   这意味着当原先的基准幅度很大（比如接近 `32767`，即有符号十六位数的极大值），此时发一条指令加上一个正向小步长，结果会跨过 `32767` 并在 `U16` 的规则下变成 `32768`。
    *   由于后续运算器将它强转为 `I16` 读取，`32768` 在有符号数中等于 **`-32768`**。**这将导致波形的幅值瞬间从正向最大跳转为负向最大！**

> [!TIP]
> **解决方案与防护建议**：永远不要依赖底层硬件的 U16 加法饱和机制！上位机在发送 Step 指令对幅度进行步进调节时，必须提前推算当前 Base 缓存的值，如果在加上 Step 参数后会越过 `32767` 或跌破 `0`，则上位机应该对下发的步进增量进行强制裁剪（Clipping）。
